# 4. Why I created labeled files

I created labeled versions of `ho_train` because I needed a final target for supervised learning.

The final target definition is:

* `unstable_15 = reversal_15 OR short_dwell_15`

Where:

* `short_dwell_15 = 1` if `dwell_s <= 15`
* `reversal_15 = 1` if the UE goes from A -> B and then B -> A within 15 seconds

Important labeled file names:

* `ho_train_100k_clean_labeled_temporal.csv`
* probably earlier labeled variants like `ho_train_100k_clean_labeled.csv` or unlabeled/labeled intermediate versions

Why I created labeled files:

* `ho_events.csv` had event timing and dwell information
* but ML training required a final binary target column
* so I added:

  * `reversal_15`
  * `short_dwell_15`
  * `unstable_15`
  * `stable_final` or similar stable indicator columns in some versions

For the final 100k labeled temporal file:

* rows = **50,997**
* `reversal_15_count = 897`
* `short_dwell_15_count = 1476`
* `unstable_15_count = 1476`
* `stable_final_count = 49,521`
* `unstable_15_rate = 0.028943`

Important note:

* in my dataset, `short_dwell_15` ended up being a **superset of** `reversal_15`
* I also checked shorter windows like 5 s and 10 s, but reversal counts there were zero, so I used **15 s**

# 6. Why I created temporal feature versions

I then extended the event-level table with **temporal delta features**.

These included:

* `delta_gap_1s`
* `delta_sinr_1s`
* `delta_distance_1s`

Reason:

* I wanted to capture whether the serving-target gap, SINR, or distance was improving or worsening **right before handover**
* this gives more temporal context than just a single snapshot

Important file:

* `ho_train_100k_clean_labeled_temporal.csv`

This file means:

* `clean`: cleaned/usable ML-ready version
* `labeled`: has final target columns
* `temporal`: includes temporal delta features

Observations:

* `delta_gap_1s` and `delta_sinr_1s` had no missing values
* `delta_distance_1s` had missing values in some smaller versions
* adding `delta_distance_1s` did not help much in performance and sometimes reduced row count because of dropped missing values

So for some final runs, I used only:

* `delta_gap_1s`
* `delta_sinr_1s`

and excluded `delta_distance_1s`

# 7. Why I explored temporal summary features

# 12. Meaning of offline gating in my project

Important concept:

* I did **offline gating**, not online deployment

Meaning:

* standard HO logic identifies candidate handover
* model assigns instability risk score to candidate event
* threshold determines whether that event would be suppressed

Formal policy used conceptually:

* execute handover if `risk <= x`
* suppress handover if `risk > x`

Important notes:

* `x` is globally fixed during offline evaluation for fair model comparison
* raw score is not shown directly in a separate final table
* the effect of score thresholding appears in the threshold-sweep / operating-point tables

This was later added to the paper to clarify how gating fits into existing handover logic.
